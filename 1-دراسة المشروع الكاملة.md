## 1. نظرة عامة

**GPMS** هو نظام Backend (API) متكامل لإدارة مشاريع التخرج الجامعية. يغطي دورة حياة المشروع بالكامل من لحظة طرحه وحتى اعتماد النتيجة النهائية.

**الأدوار المدعومة (4 أدوار):**
- **طالب (Student)**: يستعرض المشاريع، ينضم لفريق، يرفع تقارير، يشاهد النتائج
- **مشرف (Supervisor)**: ينشئ ويدير المشاريع، يقيم التقارير، يسجل الحضور
- **رئيس قسم (HOD)**: يشرف على كل المشاريع، يدير لجان المناقشة، يعتمد النتائج
- **عميد/أدمن (Dean/Admin)**: لوحة تحكم عليا، إدارة المستخدمين

**عدد الشاشات المدعومة:** 52 شاشة وظيفية موزعة:
- Auth: 4 | Student: 14 | Supervisor: 14 | HOD: 12 | Dean: 5 | Shared: 3

## 2. المشكلة والحل

### المشكلة
الجامعات تعاني من:
- عدم وجود نظام موحد لمتابعة مشاريع التخرج
- صعوبة تواصل الطلاب مع المشرفين
- ضياع التقارير بين الطرفين
- عدم وجود آلية منظمة للجان المناقشة والتقييم
- صعوبة استخراج الإحصائيات والتقارير

### الحل المقترح
نظام GPMS يقدم:
- منصة موحدة تربط الطالب والمشرف ورئيس القسم والعميد
- رفع ومتابعة التقارير إلكترونياً مع إشعارات فورية
- إدارة لجان المناقشة والتقييم بشكل رقمي كامل
- لوحات إحصائية لكل دور تظهر المؤشرات المهمة

## 3. التقنيات المستخدمة

| التقنية | الإصدار | سبب الاختيار |
|---------|---------|--------------|
| **Django** | 6.0 | إطار عمل متكامل وسريع، يوفر ORM و Admin Panel و Security |
| **Django REST Framework** | 3.16 | بناء APIs احترافية بأقل جهد، يدعم Serialization و Authentication |
| **SQLite** | - | مناسبة للتطوير الأولي، لا تحتاج خادم منفصل، مرنة |
| **Token Authentication** | - | أبسط من JWT للمشاريع المتوسطة، سهل التنفيذ |

**لماذا SQLite وليس PostgreSQL؟**
في مرحلة التطوير والاختبار، SQLite كافية وتوفر:
- لا حاجة لتثبيت PostgreSQL
- DB في ملف واحد سهل النقل والنسخ
- أداء ممتاز للمشاريع الصغيرة والمتوسطة
في الإنتاج ينصح بالتبديل إلى PostgreSQL.

**لماذا Token Auth وليس JWT؟**
- الـ Token (authtoken) أبسط في التنفيذ والفهم
- لا يحتاج Refresh Tokens
- إدارته أسهل (إنشاء/حذف التوكن)
- مناسب لنظام يستخدم نفس التوكن طوال الجلسة

```

│   ├── urls.py                     #    جميع مسارات الـ API (80+ endpoint)
│   ├── wsgi.py                     #    تشغيل المشروع على سيرفر إنتاج
│   └── asgi.py                     #    لدعم WebSockets (للمستقبل)
│
├── api/                            # 📁 التطبيق الأساسي
│   ├── models.py                   #    تعريف جداول قاعدة البيانات (11 جدول)
│   ├── serializers.py              #    تحويل البيانات من/إلى JSON (15 سيريالايزر)
│   ├── admin.py                    #    ربط الموديلات بلوحة تحكم Django Admin
│   ├── tests.py                    #    اختبارات (قابل للتوسيع)
│   │
│   ├── views/                      # 📁 كل Views APIs مصنفة حسب الدور
│   │   ├── auth_views.py           #    4 APIs: تسجيل، دخول، نسيت كلمة المرور
│   │   ├── student_views.py        #    14 APIs: الطالب
│   │   ├── supervisor_views.py     #    14 APIs: المشرف
│   │   ├── head_views.py           #    12 APIs: رئيس القسم
│   │   └── admin_views.py          #    5 APIs: العميد/الأدمن
│   │
│   └── migrations/                 #    📁 ملفات تحديث قاعدة البيانات
│       └── 0001_initial.py ...     #    16 ملف ترحيل (migration)
│
├── media/reports/                  # 📁 ملفات التقارير PDF المرفوعة
├── seed_data.py                    # 📄 بيانات تجريبية (7 مستخدمين، 3 مشاريع...)
├── requirements.txt                # 📄 الحزم المطلوبة (Django, DRF)
├── manage.py                       # 📄 سكريبت إدارة Django
└── db.sqlite3                      # 💾 قاعدة البيانات
```

### لماذا تم فصل الـ Views حسب الدور؟
- **تنظيم أفضل**: كل ملف يخص دوره واضح
- **سهولة الصيانة**: أي تعديل على دور معين يتم في ملف واحد
- **تجنب التعارض**: كل ViewSet له namespace خاص
- **القراءة**: المبرمج الجديد يفهم فوراً وين يروح لكل دور

## 5. قاعدة البيانات

### 5.1 مخطط العلاقات (Entity Relationship)

```
User (Django built-in)
  ├── 1:1 ── UserProfile (بيانات إضافية: قسم، هاتف، عنوان)
  ├── 1:N ── Project.supervisor (المشرف على المشروع)
  ├── M:N ── Project.students (فريق الطلاب)
  ├── 1:N ── Report.student (رافع التقرير)
  ├── 1:N ── Notification.recipient (مستلم الإشعار)
  ├── 1:N ── Attendance.student (حضور الطالب)
  ├── M:N ── DefenseCommittee.examiners (أعضاء اللجنة)
  ├── 1:N ── Evaluation.doctor (الدكتور المقيم)
  ├── 1:N ── JoinRequest.student (طالب طلب الانضمام)
  └── 1:N ── Department.head (رئيس القسم)

Project
  ├── 1:N ── Report (التقارير)
  ├── 1:N ── Announcement (الإعلانات)
  ├── 1:N ── JoinRequest (طلبات الانضمام)
  ├── 1:N ── Notification.project (الإشعارات المرتبطة)
  └── 1:1 ── DefenseCommittee (لجنة المناقشة)

DefenseCommittee
  └── 1:N ── Evaluation (التقييمات)

Announcement
  └── 1:N ── Attendance (سجلات الحضور)
```

### 5.2 شرح كل جدول

#### 📋 Project — المشاريع
```
title:           CharField(200)     → عنوان المشروع (مطلوب)
description:     TextField           → وصف المشروع (مطلوب)
status:          CharField(20)       → مقترح / قيد_التنفيذ / منجز (اختيارات)
supervisor:      FK→User            → المشرف (اختياري nullable)
students:        M2M→User           → فريق الطلاب (اختياري)
final_grade:     FloatField(null)    → العلامة النهائية (بعد اعتماد اللجنة)
required_reports: PositiveInt        → عدد التقارير المطلوبة (default 8)
objectives:      TextField(blank)    → أهداف المشروع
technologies:    TextField(blank)    → التقنيات المستخدمة
deliverables:    TextField(blank)    → مخرجات المشروع
```
**ملاحظة**: `on_delete=SET_NULL` للمشرف — إذا حُذف المشرف يبقى المشروع لكن بدون مشرف.

#### 📋 Report — التقارير
```
project:         FK→Project(CASCADE)     → المشروع التابع له
student:         FK→User(CASCADE,null)    → الطالب الرافع
file_title:      CharField(100)           → عنوان التقرير
file:            FileField(upload_to='reports/')  → ملف PDF
uploaded_at:     DateTime(auto_now_add)   → وقت الرفع (تلقائي)
status:          CharField(20)            → قيد_المراجعة / مقبول / بحاجة_تعديل
feedback:        TextField(null)          → ملاحظات المشرف
```

#### 📋 Notification — الإشعارات
```
recipient:       FK→User(CASCADE)     → المستلم
project:         FK→Project(null)     → المشروع المرتبط (قد يكون null)
message:         TextField            → نص الإشعار (يدعم الإيموجي)
is_read:         BooleanField(default=False) → هل قرئ؟
created_at:      DateTime(auto_now_add) → وقت الإنشاء
```
**سبب المشروع nullable**: بعض الإشعارات عامة (غير مرتبطة بمشروع معين).

#### 📋 Announcement — الإعلانات
```
project:         FK→Project(CASCADE)     → المشروع
title:           CharField(200)           → عنوان الإعلان
content:         TextField                → المحتوى
meeting_time:    DateTimeField            → موعد الاجتماع (تم التحقق أنه في المستقبل)
created_at:      DateTime(auto_now_add)   → وقت النشر
```

#### 📋 Attendance — الحضور
```
announcement:    FK→Announcement(CASCADE) → الإعلان/الاجتماع
student:         FK→User(CASCADE)         → الطالب
is_present:      BooleanField(default=False) → حاضر/غائب
marked_at:       DateTime(auto_now)        → وقت تسجيل الحضور
```
**unique_together**: (announcement, student) → يمنع تسجيل حضور مكرر لنفس الطالب.

#### 📋 DefenseCommittee — لجان المناقشة
```
project:         OneToOne→Project(CASCADE)  → المشروع (كل مشروع له لجنة واحدة)
examiners:       M2M→User                   → أعضاء اللجنة (دكاترة)
date:            DateTimeField              → تاريخ المناقشة
location:        CharField(255)             → المكان
is_finalized:    BooleanField(default=False)→ هل تم اعتماد النتيجة؟
```
**OneToOneField**: كل مشروع له لجنة واحدة فقط — هذا منطقي لأن المناقشة تحدث مرة واحدة.

#### 📋 Evaluation — التقييمات
```
committee:       FK→DefenseCommittee(CASCADE) → اللجنة
doctor:          FK→User(CASCADE)              → الدكتور المقيّم
grade:           FloatField                    → الدرجة
feedback:        TextField(blank,null)         → الملاحظات
created_at:      DateTime(auto_now_add)        → وقت التقييم
```
**unique_together**: (committee, doctor) → كل دكتور يقيم مرة واحدة فقط لكل لجنة.

#### 📋 JoinRequest — طلبات الانضمام
```
project:         FK→Project(CASCADE)   → المشروع المطلوب
student:         FK→User(CASCADE)      → الطالب
status:          CharField(20)         → قيد_الانتظار / مقبول / مرفوض
created_at:      DateTime(auto_now_add) → وقت الطلب
```
**unique_together**: (project, student) → يمنع تكرار الطلب.

#### 📋 SystemSettings — إعدادات النظام
```
max_students_per_project: PositiveInt(default=5) → الحد الأقصى للطلاب لكل مشروع
```
**نمط Singleton**: هذا الجدول يحتوي صف واحد فقط، يتم استدعاؤه بـ `.first()`.

#### 📋 PasswordResetCode — رمز إعادة تعيين كلمة المرور
```
email:           EmailField        → البريد الإلكتروني للطالب
code:            CharField(100, unique) → رمز UUID فريد
created_at:      DateTime(auto_now_add) → وقت الإنشاء
is_used:         BooleanField(default=False) → هل تم استخدامه؟
```
**صلاحية الرمز**: ساعة واحدة فقط (يتم التحقق في الـ View).

#### 📋 UserProfile — بيانات إضافية للمستخدمين
```
user:            OneToOne→User(CASCADE) → المستخدم
department:      FK→Department(null)    → القسم
phone:           CharField(20)          → رقم الهاتف
address:         TextField(blank)       → العنوان
birth_date:      DateField(null)        → تاريخ الميلاد
```

#### 📋 Department — الأقسام الأكاديمية
```
name:            CharField(100, unique) → اسم القسم
code:            CharField(10, unique)  → الرمز (مثال: IT, CS)
description:     TextField(blank)       → الوصف
head:            FK→User(null)          → رئيس القسم
created_at/updated_at: Auto timestamps
```

## 6. جميع APIs

### 6.1 🟢 Auth Module (4 APIs)

#### 1. تسجيل مستخدم جديد
```
POST /api/register/
Body:  { "email": "...", "password": "...", "first_name": "...", "last_name": "..." }
Return: { "id", "username", "email" }
Auth:  ❌ (AllowAny)
```
**التحقق**: يمنع تسجيل إيميل موجود مسبقاً.
**ملاحظة**: اسم المستخدم (username) يصبح نفس الإيميل تلقائياً.

#### 2. تسجيل الدخول
```
POST /api/login/
Body:  { "username": "...", "password": "..." }
Return: { "token": "...", "is_staff": bool, "is_superuser": bool }
Auth:  ❌ (AllowAny)
```
**التوكن**: يستخدم `rest_framework.authtoken` — يتم إنشاء توكن لكل مستخدم عند أول تسجيل دخول.
**المعلومات الإضافية**: `is_staff` و `is_superuser` تسمح للـ Frontend بتحديد صلاحيات المستخدم فوراً.

#### 3. نسيت كلمة المرور
```
POST /api/forgot-password/
Body:  { "email": "..." }
Return: { "message": "...", "code": "uuid..." }
Auth:  ❌ (AllowAny)
```
**آلية العمل**:
1. التحقق من وجود الإيميل في النظام
2. إنشاء UUID عشوائي كرمز تحقق
3. حفظ الرمز في جدول `PasswordResetCode`
4. إعادة الرمز في الاستجابة (في الإنتاج يُرسل عبر البريد الإلكتروني)

#### 4. إعادة تعيين كلمة المرور
```
POST /api/reset-password/
Body:  { "email": "...", "code": "...", "new_password": "...", "confirm_password": "..." }
Return: { "message": "..." }
Auth:  ❌ (AllowAny)
```
**التحقق**:
- تطابق new_password مع confirm_password
- الرمز موجود وغير مستخدم (is_used=False)
- صلاحية الرمز لم تنتهِ (أقل من ساعة)
- بعد النجاح: تغيير كلمة المرور، تعليم الرمز كمستخدم

**لماذا منتهى الصلاحية ساعة؟** توازن بين الأمان والمرونة — وقت كافٍ للمستخدم دون أن يكون خطراً أمنياً.

### 6.2 🟢 Student Module (14 APIs)

#### 5. استعراض المشاريع
```
GET /api/projects/
Query: ?search=كلمة&supervisor=1&status=proposed&technologies=python
Return: [ { id, title, description, status, students, required_reports, ... } ]
Auth:  ✅
```
**المنطق**:
- للطالب: يعرض المشاريع المقترحة فقط (status='proposed')
- للمشرف: يعرض مشاريعه الخاصة
- يدعم البحث (search) في العنوان والوصف والتقنيات
- يدعم التصفية حسب المشرف والحالة والتقنيات

#### 6. تفاصيل المشروع (للمشاهدة)
```
GET /api/projects/{id}/details/
Return: { title, description, objectives, technologies, deliverables, status, supervisor,
          available_seats, students, accepted_reports, required_reports, progress }
Auth:  ✅
```
**التحقق**: الطالب يشاهد فقط المشاريع المقترحة أو مشروعه الخاص.

#### 7. الانضمام لمشروع
```
POST /api/projects/{id}/join/
Body:  { "emails": ["student1@...", "student2@...", ...] }
Return: { "message": "...", "count": عدد_الطلبات }
Auth:  ✅
```
**آلية العمل**:
1. استقبال قائمة إيميلات الفريق
2. لكل إيميل: البحث عن المستخدم، إنشاء طلب انضمام (إذا لم يكن موجوداً مسبقاً)
3. إرسال إشعار للمشرف: "طلب انضمام جديد من ..."
4. المشرف لاحقاً يقبل أو يرفض

**المشرفون**: لا يمكنهم الانضمام للمشاريع.

#### 8. طلب الانضمام المعلق
```
GET /api/student/pending-request/
Return: [ { id, project_title, student_name, status, created_at, ... } ]
Auth:  ✅
```
يرجع جميع طلبات الانضمام التي قدمها الطالب (معلق/مقبول/مرفوض).

#### 9. لوحة تحكم الطالب
```
GET /student/project_dashboard/
Return: { project_title, project_description, status, accepted_reports, total_reports,
          required_reports, progress (%), supervisor }
Auth:  ✅
```
**المنطق**: يحسب التقدم عبر `(accepted_reports / required_reports) * 100`.

#### 10. رفع تقرير
```
POST /api/reports/upload/
Body:  { "file_title": "...", "file": (PDF), "project": id }
Return: التقرير المنشأ
Auth:  ✅
```
**التحقق**: الطالب يجب أن يكون ضمن فريق هذا المشروع.
**بعد الرفع**: إرسال إشعار للمشرف تلقائياً.

#### 11. تقاريري
```
GET /my-reports/
Return: [ { id, file_title, file, status, feedback, student, project } ]
Auth:  ✅
```
**المنطق**: يعرض جميع تقارير المشروع الذي ينتمي إليه الطالب. كل فريق المشروع يشاهد نفس التقارير.

#### 12. تفاصيل التقرير
```
GET /api/reports/{id}/
Return: { id, file_title, file_url, status, feedback, uploaded_at, student, project_title }
Auth:  ✅
```
**التحقق**: فقط أعضاء فريق المشروع أو المشرف يمكنهم المشاهدة.

#### 13. الإشعارات
```
GET /my-notifications/
Return: [ { id, message, is_read, created_at } ]
Auth:  ✅
```

#### 14. الإعلانات
```
GET /api/announcements/summary/
Return: [ { id, title, project_title, content, meeting_time, created_at, student_count, attendance_count } ]
Auth:  ✅
```
**لاحظ**: هذا الـ API مشترك بين الطالب والمشرف — الطالب يرى إعلانات مشروعه، المشرف يرى إعلانات كل مشاريعه.

#### 15. تفاصيل الإعلان
```
GET /api/announcements/{id}/details/
Return: الإعلان + الحضور (للمشرف كل الطلاب، للطالب حالته فقط)
Auth:  ✅
```
**مشارك**: بين الطالب والمشرف.

#### 16. ملخص الحضور
```
GET /api/student/attendance-summary/
Return: { total_meetings, present, absent, attendance_rate (%) }
Auth:  ✅
```

#### 17. لجنتي
```
GET /api/student/my_committee/
Return: { project_title, date, location, examiners, is_finalized }
Auth:  ✅
```

#### 18. النتيجة النهائية
```
GET /api/student/final_result/
Return: { project_title, final_grade, status }
Auth:  ✅
```

### 6.3 🟢 Supervisor Module (14 APIs)

#### 19. لوحة تحكم المشرف
```
GET /supervisor/dashboard/
Return: { stats: { total_projects, total_students, pending_reports, avg_attendance },
          recent_activity: [...], my_projects: [...] }
Auth:  ✅
```
**الإحصائيات**: 4 KPIs + آخر 5 نشاطات + قائمة المشاريع.

#### 20. إنشاء مشروع
```
POST /supervisor/create-project/
```

## 7. نظام الصلاحيات

### 7.1 مستويات الصلاحية في Django
```
User
├── is_staff = False, is_superuser = False  → 👤 Student
├── is_staff = True,  is_superuser = False  → 👨‍🏫 Supervisor
├── is_staff = True,  is_superuser = True   → 👨‍💼 HOD / 👨‍💻 Dean
└── Groups → 'Dean'                         → 👨‍💻 Dean (صلاحية إضافية)
```

### 7.2 كيف تتحقق APIs من الصلاحية؟

**على مستوى الـ View (Method 1):**
```python
# في student_views.py — شرط بسيط
if request.user.is_staff:
    return Response({'error': 'المشرفين لا يمكنهم الانضمام للمشاريع'})

# في supervisor_views.py
if not self.request.user.is_staff:
    raise PermissionDenied("هذه الصلاحية للمشرفين فقط")
```

**على مستوى الـ Permission Class (Method 2):**
```python
# في admin_views.py
class IsDean(permissions.BasePermission):
    def has_permission(self, request, view):
        return request.user.groups.filter(name='Dean').exists()
```

**على مستوى الـ queryset (Method 3):**
```python
# في supervisor_views.py — يرى مشاريعه فقط
def get_queryset(self):
    return Project.objects.filter(supervisor=self.request.user)
```

## 8. سير العمل (Workflow)

### 8.1 دورة حياة المشروع
```
مقترح (proposed)  ────→  قيد التنفيذ (in_progress)  ────→  منجز (completed)
       │                        │                              │
       │ ← مشرف ينشئ            │ ← تم قبول العدد الكافي       │ ← تم اعتماد النتيجة
       │ ← طلاب يشوفوه          │   من الطلاب                   │   من رئيس القسم
       │ ← طلاب يقدموا طلبات    │ ← الطلاب يرفعوا تقارير        │
       │                        │ ← المشرف يقيم                 │
       │                        │ ← فيه إعلانات وحضور           │
       └────────────────────────┴──────────────────────────────┘
```

### 8.2 رحلة الطالب الكاملة
```
1. يسجل حساب (register)
2. يتسجل دخول (login) → يستلم token
3. يتصفح المشاريع المقترحة (browse)
4. يشوف تفاصيل مشروع عاجبه (details)
5. يقدم طلب انضمام مع فريقه (join)
6. يستنى موافقة المشرف
7. بعد القبول: يشوف Dashboard مشروعه
8. يرفع تقارير دورية (upload report)
9. يتلقى إشعارات بالتقييم
10. يشوف إعلانات الاجتماعات
11. يسجل حضور (يدوياً من المشرف)
12. يشوف تفاصيل لجنة المناقشة
13. يستلم النتيجة النهائية
```

### 8.3 رحلة المشرف الكاملة
```
1. يسجل دخول
2. ينشئ مشروع (مقترح)
3. يستقبل طلبات الانضمام
4. يقبل الطلاب المناسبين
5. يتابع Dashboard المشرف (إحصائيات)
6. يستقبل التقارير → يقيمها
7. ينشئ إعلانات اجتماعات
8. يسجل حضور الطلاب
9. يشارك كلجنة مناقشة
10. يقيم المشاريع الموكلة له
```

### 8.4 رحلة رئيس القسم
```
1. يسجل دخول
2. Dashboard يعرض إحصائيات القسم
3. يراقب كل المشاريع
4. ينشئ لجان المناقشة
5. بعد التقييمات → يعتمد النتيجة
6. يضبط إعدادات النظام (الحد الأقصى)
7. يدير الأقسام
```

## 10. الاعتبارات الأمنية

### 10.1 المصادقة (Authentication)
- جميع APIs تتطلب `Token Authentication` ما عدا `Register`, `Login`, `Forgot/Reset Password`
- التوكن يُنشأ عند أول تسجيل دخول ويبقى صالحاً
- كل طلب API يجب أن يحتوي Header: `Authorization: Token <token_key>`

### 10.2 التفويض (Authorization)
- كل API يتحقق من صلاحية المستخدم قبل السماح بالعملية
- التحقق يكون على 3 مستويات:
  1. **is_staff**: للمشرفين
  2. **is_superuser**: لرئيس القسم والعميد
  3. **مجموعة (Group)**: للعميد فقط (`IsDean`)
- بالإضافة إلى التحقق من ملكية المشروع (المشرف يدير مشاريعه فقط)

### 10.3 حماية البيانات
- كلمات المرور: تُشفّر تلقائياً عبر `User.set_password()`
- الملفات: تُرفع إلى مجلد `media/reports/` معزول عن الكود
- **لا يتم تسريب**: كلمات المرور لا تُعاد أبداً في الاستجابات
- رمز إعادة التعيين: له صلاحية ساعة واحدة فقط، ويُستخدم مرة واحدة

### 10.4 التحقق من صحة البيانات
- التحقق من الإيميل المكرر عند التسجيل
- التحقق من تاريخ الاجتماع (لا يمكن أن يكون في الماضي)
- التحقق من عدم تجاوز الحد الأقصى للطلاب
- التحقق من عدم تعديل/حذف مشاريع قيد التنفيذ

### 10.5 نقاط ضعف محتملة
- **لا يوجد rate limiting**: يمكن إرسال عدد لا نهائي من الطلبات
- **لا يوجد https**: في الإنتاج يجب تفعيل SSL
- **لا يوجد CSRF للـ API**: يستخدم Token Authentication
- **لا يوجد Email Verification**: الحسابات تنشط فوراً

## 11. الحالات الخاصة (Edge Cases)

### 11.1 ماذا لو...
**1. طالب قدم طلب انضمام لمشروع ثم المشرف حذف المشروع؟**
الطلب يبقى في قاعدة البيانات لكن المشروع غير موجود → `CASCADE` يحذف الطلبات تلقائياً.

**2. مشرف حاول إضافة طالب يتجاوز الحد المسموح؟**
التحقق في `ApproveJoinRequestView`: إذا `students.count() >= max_students_per_project` ← يمنع ويظهر رسالة خطأ.

**3. طالب رفع تقرير لمشروع ليس ضمن فريقه؟**
التحقق في `UploadReportView`: `project.students.filter(id=user.id).exists()` ← يمنع.

**4. مشرف حاول تقييم تقرير من مشروع ليس له؟**
`get_queryset` في `SupervisorEvaluateReport` : `Report.objects.filter(project__supervisor=self.request.user)` ← يرى تقارير مشاريعه فقط.

**5. رئيس القسم أنشأ لجنة بدون مشرف المشروع؟**
التحقق في `DefenseCommitteeViewSet.perform_create`: يجب أن يكون المشرف ضمن `examiners`.

**6. لا يوجد إعدادات نظام بعد (SystemSettings)?**
`SystemSettings.objects.first()` يرجع None → التحقق موجود في `ApproveJoinRequestView` ويعيد خطأ.

**7. طالب ليس لديه مشروع بعد؟**
`StudentProjectDashboardView`: يرجع `{'error': 'لا يوجد مشروع مسجل لك'}`.

**8. مشروع ليس له لجنة مناقشة بعد؟**
`StudentMyCommitteeView`: يرجع `{'error': 'لم يتم تحديد لجنة مناقشة بعد'}`.

**9. دكتورين أعطوا تقييمين مختلفين جداً؟**
يتم حساب المتوسط الحسابي `Avg('grade')` في `finalize_result`. إذا الفرق كبير، رئيس القسم يقرر.

**10. انتهت صلاحية رمز إعادة التعيين؟**
`ResetPasswordView` يتحقق: `reset_code.created_at < timezone.now() - timedelta(hours=1)` ← يطلب رمز جديد.
